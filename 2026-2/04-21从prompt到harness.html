从 prompt 到 harness

关于提示词工程的演进。

提示词工程 prompt engineering，上下文工程 context engineering，驾驭工程 harness engineering，概念一个接一个，但其实我是觉得很 trivial。但 trivial 的东西中涌现出让人惊掉下巴的魔法，LLM 发展历程自己就已经给我们展示过了……

按我自己的理解，harness 其实就是自动化地监督 AI 的行为（比如读一个文件太多次就用提示词去警告它，比如代码实现后强制性的检查流程），自动化地改进 AI 的上下文，比如分析过去的对话，找出错误，得出总结，并记录下来。

Prompt vs Context

从结构上做简单区分的话，提示词工程，就是通过静态去构造单条提示词（主要指的是首条系统提示词）去约束模型的行为，比如告知 AI 你是 Python 专家，你是 AI 猫娘……然后用户随心所欲去输入啥东西，即这条提示词以外的，剩下的对话历史就全是用户和 AI 的对话；而上下文工程，则是动态去构造整个上下文而非是仅仅单条提示词都动态去构建，SillyTavern 是这方面的典型，Naive RAG 也是类似。

上面的静态和动态，可以说是从 LLM 客户端来看的。静态，指的是 LLM 客户端在开始对话前只需构造这个系统提示词一次，之后客户端便不再去执行业务逻辑，而是单纯地服务于用户和 AI 的对话，去不断地扩展当前上下文；而动态，指的则是 LLM 客户端每次都会完全重新拼接，构造整个上下文，比如 Naive RAG，SillyTavern 中的绿灯世界书……

这里的重点在于，上下文工程将上下文去视为一个动态的，可灵活构造的东西，而提示词工程则是将上下文视作单纯的 chat history。

从行为上来说，提示词工程则是对单条提示词去进行修修补补，去让这条上下文更好地起作用，而上下文工程则是调整整个上下文的构造方式。

但上述论述其实没啥意义，提示词工程和上下文工程，其实更多是心智上的说法，其实一切都是上下文，问题在于我改的是某一段 prompt 还是 prompt 之间的关系。

不过按我说，智能体编排也可说是上下文工程——只是我在构建这个上下文的时候顺便也调了其他 agent 而已。

Harness 之前的问题

参考了 https://www.bilibili.com/video/BV1C9QuBZEKz。

A 社的工程师总结的常见三个问题：

  1. One-Shot：试图在一次 session 中解决问题，结果就是上下文耗尽（而压缩带来心智上的损耗），重启时因为文档缺失 AI 只能靠猜，方向走歪
  2. Premature Completion：完成核心功能后 AI 就开摆说做完了，todo-list 都给你勾上
  3. False Completion：错误地标记功能完成，不测试不验证

还有一个风险——AI 会忠实地复制原有代码库里的代码风格、模式，无论好坏，这会快速积累技术债。

Harness 试图解决这些问题。

Harness engineering

其实 Harness engineering……我怎么看它似乎都该说是一种思想，是属于上下文工程的，它不能拔高到上下文工程这个层级。

Harness 分为 4 部分：

  1. Context Engineering，该说是一个指导思想吧，要求指令不能太多以挤占文档，代码等内容的注意力，它提出 AGENTS.md 要小巧，最好小于 60 行，AI 去按需检索。
    1. 这里指的应该是优化提示词……提示词里要专门写模型不会的东西，模型会的不要重复
  2. Architecture Constraints，架构约束，最核心的部分，用确定性去取代模糊指令。这里具体指的是去做一个严格的分层架构——Types->Config->Repo->Service->Runtime->UI，绝不能反向依赖，而且这里使用的是 linter 去机械化地做而非使用 Prompt。约束永远比指令更有效
  3. Feedback Loop，代码审查，Agent 审 Agent，在 AI 完成编写后回应之前（Checklist 拦截）让它自己去进行检查，Build & Self-Verify。规划、构建、（对照最初需求）验证、修复。（这 TM 不是 PDCA 吗）
  4. Entropy Management，熵管理，OpenAI 称为 GC——定期用 Agent 扫描文档、架构、代码风格不一致并进行修改

LangChain 提出了更多实操建议：

  1. Trace Analyzer，提出这个 skill 去让 AI 能去分析过去的上下文去找到 AI 犯的错误并记录下来
  2. LocalContextMiddleware……将执行环境的上下文准确丢给 AI
  3. LoopDetectionMiddleware……操作同一个文件过多次后在上下文中提醒 AI 换方法

Vercel 提出精简工具——更少的工具会更少地混淆 AI。

行吧，没有新东西，但确实是有了新 idea，我觉得我得尝试去用 dify 或者 langflow 去搭点什么东西才能有更多感性经验去用来理解它。

视频中提到，OpenAI 工程师说，每当你发现 Agent 犯了一个错误，你就花时间去工程化一个解决方案，让它再也不会犯同样的错，这话确实不赖，得 mark。
