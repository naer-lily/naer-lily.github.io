学习 Python 的多进程和 IPC 机制

Python 的 IPC 相关的东西以前没接触过，最近碰到一个需求，就是想用 MCP 去包装 botpy（QQ 官方的机器人框架），这让有必要去研究一下 Python 的 IPC 以让我能看懂它们究竟是咋交互的。实际上 Python 中很多框架都是直接阻塞当前线程的，而又有很多东西，比如展示托盘，GUI，它们的业务逻辑需要运行在主线程中，这就使得在使用多个框架的时候，多进程成为必要的东西。

使用 FastMCP 和 botpy

这个是我的一个现实需求了：

  1. botpy 底层跑一个事件循环，有心跳、回调机制，但同时也允许主动触发发送消息
  2. FastMCP 底层跑一个事件循环，我需要暴露出一个工具去触发发送消息

我测试了一下，在同一个进程中时，莫名其妙 botpy 不太工作，原因未知（我盲猜是 botpy 内部的实现问题，可能有莫名其妙的忙等，配合 GIL 让所有线程全都饿死，连 Ctrl+C 都不能让它退出来），但我让 AI 去改成多进程后就能工作了。

这个实现其实很 trivial——父进程创建一个 Queue（多进程共享，原理后面写），这个 Queue 大小为 0，即它是一个 Java 语境中的SynchronousQueue，生产者必须等待消费者当场取，消费者必须等待生产者当场给。这个 Queue 大小为无穷（Python 中没有 Java 语境下的SynchronousQueue），用这个 Queue 去做一个单向管道，将主进程的请求发给子进程。

那问题来了——双向呢？如果想要一个返回值呢？这时候明显就需要使用两个 Queue 了，而且需要某种同步手段；最简单的方法是使用某种锁去保证生产者先发后取（原子地），消费者先取后发（原子地），但这个是单向的，某种程度上来说是同步的，复杂点就可以做一个异步的，比如每个请求都做个标识，在响应中找到标识后将响应导回原本的请求者……当然，我没看过工程实践具体该咋整。

创建 Process 时发生了啥

Process（后面称子进程），有三种启动方法——fork，spawn 和 forkserver，这里只说 fork 和 spawn。

  • fork：linux 的默认启动方法，仅限、unix，利用操作系统提供的 fork 函数实现子进程。它会 CopyOnWrite 地复用父进程的一切资源然后执行自己的逻辑。*fork 不推荐使用！因为它只会复制当前线程的资源（Python 虚拟机是多线程的，第三方依赖也经常引入自己的线程），且同步工具如锁的状态也会被复制，这极容易造成后台的死锁等问题！
  • spawn：windows 和 macos 的默认启动方法，真正地创建新的 Python 解释器，使用模块导入的方式（因此执行当前文件时，__name__不是__main__）导入当前代码并执行操作。spawn 是更推荐的。

这里不提 fork（fork 的好处是性能更高，启动更快，不需要序列化反序列化，但我认为这个好处可以由进程池去替代），建议工程代码中始终使用 spawn 放在最前头：

后面的学习也以 spawn 为主。

启动子进程时，注意到会传递给子进程一个函数和相应参数，这个函数和这些参数会被 pickle 序列化，然后在子进程处反序列化并进行调用。这时候就要问了——既然函数也序列化了，为何 spawn 还需要导入这个模块？原因在于，函数的 pickle 序列化只是序列化成一个引用该函数的路径（如__main__.hello）而不包含函数逻辑。因此匿名函数和非顶层的函数都无法被序列化，而子进程需要 import 这个模块，并根据这个函数引用和参数去执行这个函数。

子进程需要使用模块导入该模块，这要求父进程的创建子进程的代码必须在if __name__ == '__main__'中执行，不然的话就会导致无限循环了（实际上会直接报错）。

IPC 同步工具

pickle 序列化，这才是魔法所在之处——多进程模块中，提供了诸多同步工具用于主进程和子进程进行交互。它们的 pickle 序列化都经过设计——在反序列化前后会引用同一个文件描述符或句柄或随便叫什么名字，总之是操作系统级别的，使得其反序列化前后可以认为是同一个对象，使得不同进程可以用其进行交互。

这些同步工具和多线程的工具类似，也是老熟人了（毕竟都合了并发编程理论），但问题在于，多线程里的各种同步原语可以说是 Java 或 Python 虚拟机自己抽象的，和操作系统交互极少，主要是在用户态，而多进程同步必须依赖操作系统，在内核态里打转，非常昂贵。

这里只是列一下（底层的同步原语我大概是没有机会去用到的）。

首先是数据传输类：

  • Queue：多生产者多消费者队列，线程安全+进程安全，可以 join
  • SimpleQueue：简化版的 Queue，只有 get 和 put，性能高，无界
  • Pipe：双向管道，Queue 底层其实也使用了 Pipe
    • 一个问题是 Pipe 很底层——它是操作系统的管道的直接包装，它不线程安全（对某一端的连接的并发 send 或并发 recv 会导致乱码（两端连接同时 send 或 recv 则线程安全，即全双工）），有缓冲区大小限制（64KB 似乎）。如果并非性能有极端需求，还是直接用俩 Queue 吧。

然后是同步原语类：

  • Lock，RLock：互斥锁和可重入锁，just like old times
  • Condition：wait-notify 模式
  • Semaphore：信号量，经典了（计数器，acquire-1（到 0 阻塞），release+1）
  • Event：布尔标志位，set-wait（wait 等待该标志位为真，set 设置该标志位为真，clear 设置该标志位为假）

最后是共享状态类（使用内存映射文件之类的技术）：

  • Value：单个值
  • Array：多个同类型值
  • Manager：使用代理模式共享复杂对象

但说到底学上面的这么多东西很大程度上是没啥意义的 hhh，我感觉我如果平常只是造轮子，比如把 pystray 和 botpy 和 fastMCP 用多进程组到一起这样，我很可能只需要实现简单的 RPC 就行了。

我们应当关心的是具体问题——如何调用一个同步的进程的一个方法得到返回值，和如何调用一个异步的进程（即主线程跑事件循环）的一个方法，去得到返回值。而这时候具体该怎么办呢……？而其实上面的例子就直接有了——直接上 Pipe 或 Queue。
