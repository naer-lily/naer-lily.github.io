Java 自己

HashMap 底层

HashMap 是拉链表——首先是数组+链表的结构；在某个特定槽位的链表大小到达一定后，这个槽位的链表转变为红黑树结构。

HashMap 初始容量为 16，自定义初始容量时，容量会被调整为大于等于它的最小二次幂，如配置初始容量为 10 时实际容量为 16。这个初始容量指的是 bucket 的数量，因为这里的限制，bucket 的数量永远是二的幂。

关于扩容，扩容发生在总大小大于扩容阈值时，扩容阈值为 加载因子 * capacity，默认加载因子为 0.75时。

扩容要求重新进行哈希，Java 8 有过一些优化，但重新哈希总的来说时必要的。

关于转树，当某个链表 bucket 大于 8 的时候，这个链表会被转换成红黑树。但这里有一个特殊情况——当哈希表当前容量小于 64 的时候，会优先执行扩容而非转树。

HashMap 索引时，先通过 HashCode 去哈希找到 bucket，再通过 equals 判等。

IdentityHashMap，则通过System.IdentityHashCode去哈希找到 bucket，再通过==去判等。其中System.IdentityHashCode可以认为是一种对对象本身标识符的哈希（可能会碰撞）。

LinkedHashMap——等于是一个 HashMap 加上一个双向链表，HashMap 索引到链表，链表再索引到实际元素，每次访问操作时都要修改这个双向链表来始终维持 LRU 的这个访问顺序。

Synchronized 锁升级

Java 的每个对象都持有一个锁供我们使用 synchronized 块去使用（一个有趣的手段是通过String.intern 去按字符串作为 ID 去获得锁的实例，等于是把字符串池当成锁池了）。

但这里不说这个。总之 Java 做了优化，保证在无竞争、轻度竞争的情况时，让使用锁尽量高效。锁从轻量级到重量级称为锁的升级，锁的升级是不可逆的。

每个对象头都有一个 Mark Word，它包含锁标志位和相应内容：

  • 无锁：表示未锁定，可偏向
  • 偏向锁：通过 CAS 将当前线程 ID 写入 Mark Word，后续重入、退出时不需要任何同步操作
    • 偏向锁明显适用于轻竞争的情况——主要只有一个线程在玩花活，它根本就不需要做任何同步操作。
    • 在一个线程（原线程）持有偏向锁，而另一个线程（竞争线程）尝试获取偏向锁的时候，这时候原线程在安全点暂停，锁升级成轻量级锁，原线程仍旧持有锁，竞争线程自旋
    • JDK 15 之后偏向锁默认禁用，上来就是轻量级锁，因为现代高并发应用中偏向锁反而更昂贵
  • 轻量级锁：在当前线程栈帧中创建一个 Lock Record 去备份……blablabla，总之，这时候竞争线程仍旧是自旋
  • 重量级锁：对应操作系统级别的互斥量 Mutex，内核态直到这时候才出场。

其实锁的升级就是一个从假装没有锁（偏向锁）到乐观锁到悲观锁，从用户态自旋到内核态阻塞的过程，低并发时，乐观锁，用户态自旋性能更好，高并发时，悲观锁，内核态阻塞性能更好。

ConcurrentHashMap

我不学实现，只要知道它的常用 API 和使用 pattern 即可。

ConcurrentHashMap 即线程安全的支持高并发的 HashMap。它的存储的数据结构同 HashMap，但通过 CAS+synchronized 去实现了线程安全，同时保证写操作只锁一个桶，读操作无锁，并发扩容。

ConcurrentHashMap 包含 HashMap 的常用操作，但它提供了很多 CAS 操作以原子地进行某种操作。如：

  • putIfAbsent：不存在时放入
  • remove(k, v) ：只有 v 匹配时才删除 k
  • replace(k, oldV, newV)：只有 v 匹配时才替换
  • computeIfAbsent，不存在时计算和放入
  • computeIfPresent：存在时计算和放入（返回 null 时删除）
  • compute：计算和放入（可能不存在，返回 null 时删除）
  • merge(k, newV, mergeFn)：key 不存在时直接放入 newV，否则通过 mergeFn 执行合并后放入

ConcurrentHashMap 支持并行的迭代，但迭代时是弱一致性，可能看到过期数据。

关于 ConcurrentHashMap 使用的常见 pattern：

  • 线程安全的本地缓存：直接 computeIfAbsent 去原子地设置缓存
  • 多线程计数器：value 直接用基础类型或 Atomic 类型或 LongAdder

注意——ConcurrentHashMap 不支持 null 的键和值！

ConcurrentHashMap 偶尔仍旧需要外部的同步逻辑，这时候使用外部的锁，不要锁 ConcurrentHashMap 的实例。

volatile，DCL

volatile 禁止指令重排序，保证多线程下的可见性——通过内存屏障保证对它的修改马上写回主存并保证其他线程从主存加载数据，一个线程修改了一个 volatile 的值，其他线程马上就能看到改变。但 volatile 不保证原子性。

DCL 即双重检查锁，必须依赖 volatile 变量，以保证对变量的设值马上被看见，且避免指令重排。

关于避免指令重排，比如懒汉式单例中，单例可能还未初始化完成时就已经赋值了（new 有三步，赋值可能出现在实际初始化之前——注意外层的 if 可是不在锁里面的！），导致其他线程误以为初始化完成。

DCL 的模式很简单——我先检查是否满足，不满足我就先锁，但我检查和我锁这两部分的间隔里，可能已经有人先动过手脚了，而在我锁了之后没人能动手脚，所以锁完后我得再检查一下当前是否满足。

为什么要双重检查锁？因为要避免无意义的锁定，在真正要执行操作的时候才去进行锁定。

我在两个场景下使用过 DCL：

  • 懒汉式单例：判空-锁-判空-初始化
  • 在进行一个远程 HTTP 调用前检查是否需要更新 access token：判断是否需要更新-锁-判断是否需要更新-获取和设置 token

当然，单例模式还有静态内部类的写法，利用 JVM 的内部类延迟加载的方式（IODH，Initialization-On-Demand-Holder），这个我也使用过。

ThreadLocal

ThreadLocal，其实就是线程特定的全局变量，常用在设置 session 信息中，比如 Spring MVC，底下的 Filter 看请求来了读取用户信息然后保存到 ThreadLocal 变量中供后续客户端代码使用，偶尔也可以真的在自己的业务中拿来当全局变量——在多个函数中共享一个参数。

ThreadLocal 本身不存储值，它只是一个引用 ThreadLocalMap 的工具类，而这个 ThreadLocalMap，它存在于 Thread 对象中。ThreadLocal 用自己当 key 去从 ThreadLocalMap 中做索引。

ThreadLocalMap 对 ThreadLocal 是弱引用的，这让 ThreadLocal 可以被 GC 掉——显然这发生在使用局部的 ThreadLocal 的时候。但 ThreadLocal 所存储的值是被Entry->ThreadLocalMap->Thread强引用的，所以只要 Thread 还活着它存储的值就是持久的。这会出现问题——Spring MVC 中，处理用户请求的是一个线程池，某个线程在上个请求中如果没有清空数据，会导致下一次请求时仍看到这个数据，所以必须 remove。

JVM 内存区域和 GC 和 JVM 调优

关于内存区域，全局的有——堆，元空间，直接内存；线程本地的有栈，PC 等。

简单来说，堆就是 Java 的对象实际存放的地方，元空间则包含各种静态的元数据，常量池（只存符号引用和基础类型的字面量）等；直接内存是 NIO 编程中直接分配的内存块；线程的栈则是函数调用栈，实际上还有一个 native 方法栈。

包装对象的缓存池，和字符串常量池（来自String.intern），都存在堆中。元空间的常量池为每个类的常量池在运行时的表示，它只存储符号引用和基础类型的字面量，倘若变量引用的是对象，则对象本身还是在堆里。

但这里不需要拎那么清楚。

关于 GC（又来啦……）。

从 GC 的视角来看，堆不再是一块单纯的大内存，而是一套按对象存活时间来划分区域的回收系统。

GC 有一个核心假设——绝大部分对象都是朝生夕灭，那些活得久的对象将会活得久（需要回收的可能性更低）。

所以 GC 将堆进行所谓的分代——年轻代（Young Gen）和老年代（Old Gen），年轻代高频 Minor GC，存放新对象，老年代低频 Full GC，活得久的对象会被移动到老年代。

而年轻代进一步细分——Eden 和 Survivor；Survivor 是双缓冲区结构，两块缓冲区分别叫 from 和 to（相对的名字）。新对象分配到 Eden，每次 Minor GC 时，Eden 和 from Suvrivor 复制到 to Survivor，然后两个 Survivor 角色互换。

Survivor 的双缓冲保证不产生内存碎片（每次 GC 都是直接将全部活对象都放到 to Survivor，而非是简单地从一块内存中清掉一部分，留下一个孔），而且速度够快（年轻代存活的对象一般少），代价是空间浪费一倍。

一个对象的一生——分配后进入 Eden，Minor GC 时在 Survivor 区来回倒腾；年龄超过一个阈值后送入老年代，老年代满后做 Full GC。

进行 GC 的那个实体叫所谓收集器。一般来说，收集器对 Minor GC（年轻代）的 GC 算法是复制，Full GC（老年代）的 GC 算法是标记-整理。这是因为：

  • 年轻代存活率低，超过 90%的对象可回收，复制代价低且无碎片
  • 老年代存活率高，复制不划算，但需要解决碎片问题

现代 GC 算法则主要是在逻辑上分代，物理上是混合的，如 G1 算法，这个就不表了。

关于 JVM 调优，JVM 调优的主要目的是减少 Full GC 的频率，次要的则是降低 GC 停顿时间，提高吞吐量……

ClassLoader

双亲委派——子加载器加载的时候优先委托父加载器去加载，父加载器不存在或找不到类定义的时候，才让子加载器去加载。其实就是父加载器优先。

为何要双亲委派？为了安全——Java 内置的各种类要让父加载器去加载以避免被用户自定义的加载器所篡改，另一个原因是避免重复加载：子加载器不能加载与父加载器所加载的同名的类型，如java.lang.String，这个同样也可以说是防篡改。

Spring 底层

Autowired 和 Resource

Autowired 是 by type 注入，Resource 是 by name 注入。

实际使用时它们很多时候是可以互换的——Resource 名字没找到会退化为类型查找；而对某个类型有多个 Bean 对应时，这时候才会有部分的抽象泄漏，此时 Autowired 需要加 Qualifier，而 Resource 可以用变量名字去匹配。

反正平常一直用 Autowired 就行了。

Bean 生命周期

我们一般使用的都是单例 Bean。Bean 一般的生命周期是：

  1. Spring 容器时加载 BeanDefinition，仅仅是注册定义
  2. 实例化原始对象，一般通过反射调用构造器、Configuration 的 @Bean 方法，然后 FactoryBean。
  3. 依赖注入（必须注入代理后的对象）
  4. 各种 Aware 接口
  5. BeanPostProcessor 前置处理
    1. BeanPostProcessor 是自定义的前后置 Bean 处理器，postProcessBeforeInitialization 是前置处理
    2. CommonAnnotationBeanPostProcessor 在此处处理 @PostConstruct 方法
  6. 初始化，调用 afterPropertiesSet()，然后 init-method
  7. BeanPostProcessor 后置处理
    1. postProcessAfterInitialization 是后置处理
    2. AOP 代理在此时完成
  8. 丢到单例池中（标志着 Bean 就绪）
  9. Bean 销毁时调用 destroy

三级缓存和循环依赖

Bean 的初始化过程简述是：

  1. 实例化原始对象
  2. 依赖注入（必须注入代理后的对象）
  3. @PostConstruct 方法
  4. 创建 AOP 代理

这个生命周期要求使用三级缓存，以保证只有循环依赖，即这个生命周期无法解决的情况时，能在循环依赖的部分临时地违背这个生命周期。

最佳答案——避免循环依赖，循环依赖很坏！除了内部调用时依赖自己，这个没啥可说的。要注入自己，使用 Lazy 注解或使用 ObjectProvider，或在运行时通过 SpringUtils 之类的方法去动态获取。

Spring 初始化时有三级缓存，解决单例 bean 循环依赖的问题（面试考这个是否有毛病……）。

显然——循环依赖时只能通过字段和 setter 注入，构造器注入是不能的。

倘若没有 AOP，其实只需要两级缓存——一级存成品（依赖已经注入完成），一级存半成品（PostConstruct 未调用……），比如 A，B 互相依赖，这时候 A 和 B 都先放到二级缓存，然后互相注入对方后再放到成品……可惜我们有 AOP，而且

注意——AOP 代理是在依赖注入之后创建的，而依赖注入必须要求注入代理后的依赖，这个矛盾两级缓存解决不了：

  1. 实例化 A，将 A 丢进二级缓存
  2. 依赖注入 A，发现要注入 B（的代理）
    1. 实例化 B，将 B 丢进二级缓存
    2. 依赖注入 B，发现要注入 A 的代理（此时还未创建）
    3. 我们退出来重新实例化 A 并过早地先生成 A 的代理？（不存在的，算法太复杂）

而三级缓存时，我们实际上不是先实例化 A，而是先创建一个生产 A 的工厂，提供 A 的生成的功能——三级缓存时，A 其实并没有实际实例化出来：

要走三级缓存，需要设置 allowEarlyReference=true，这个参数就是允许循环依赖的参数。

  1. 实例化 A，将 A 的工厂丢进三级缓存
  2. 检查 A 的依赖，发现要注入 B（的代理）
    1. 实例化 B，将 B 的工厂丢进三级缓存
    2. 依赖注入 B，发现要注入 A（的代理），而 A 在三级缓存里
    3. 据此执行获取 A 的逻辑——工厂 A 根据 A 的实例并提早创建 AOP 代理，将 A 的代理换到二级缓存
    4. 将 B 丢到二级缓存（注意 B 的工厂从未被使用过）
    5. 将 A 注入 B，执行 B 的后处理（PostConstruct 和 AOP），得到 B 的成品，换到一级缓存
  3. 将 B 注入 A，执行 A 的剩余后处理（A 已经提早创建代理了，现在要执行其他处理），将 A 的代理换到一级缓存

明显可以发现——A 和 B 循环依赖时，A 或者 B 在自己初始化时必然其中某一个会看到没有完全初始化完成的对方。

Spring Boot 自动配置

导入一个新 jar 包时，在包名不匹配扫描配置的时候，Spring Boot 是怎么动态加载知晓它和加载它的？

首先启动类要标上 EnableAutoConfiguration 注解以让 Spring Boot 这套花活起起来。

通过 jar 包的resources/META-INF/spring.factories，它是一个 properties 格式文件，Spring 会自动加载所有 jar 包下的该文件的org.springframework.boot.autoconfigure.EnableAutoConfiguration，这个字段保存为 Spring 需要去加载的一系列 Configuration 类，Spring 会实例化它们。

AutoConfiguration 本质上仍旧是 Configuration。

Configuration 支持条件装配，一系列 ConditionalOn 注解等，我记得我玩过一些花活在 @Bean 里做判断。
